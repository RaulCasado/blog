Antes vs. Después

El mismo problema, dos arquitecturas opuestas.

Antes

Todo en el cliente, filtrando el DOM

Se renderizaban todos los posts en una sola rejilla. El cuerpo completo de cada post (Markdown renderizado) se volcaba en atributos data-content. El "buscador" recorría las tarjetas y hacía display:none según un String.includes() de substring.

  • ✕HTML hinchado: el documento llevaba el texto íntegro de todos los posts, los buscases o no.
  • ✕DOM pesado: N tarjetas siempre montadas; el scroll móvil era una lista infinita.
  • ✕Búsqueda rígida: substring exacto, sin tolerancia a erratas ni ranking de relevancia.
  • ✕HTML sin sanear en atributos y trabajo de filtrado proporcional a todo el corpus en cada tecla.
  • ✕Sin paginación: no escalaba a medida que crecía el número de artículos.

Después

Paginación estática + índice fuzzy lazy

Dos modos que no compiten. Navegar: páginas estáticas de 9 posts (slice en build). Buscar: al enfocar el input se carga fuse.js + un JSON de metadatos y se hace búsqueda difusa sobre todo el corpus, no sobre lo pintado.

  • ✓HTML ligero: cero cuerpos de post en el índice; solo 9 tarjetas por página.
  • ✓Carga diferida: quien solo navega no descarga ni Fuse ni el JSON.
  • ✓Fuzzy + ranking: tolera typos y ordena por relevancia con pesos por campo.
  • ✓Render seguro: resultados construidos con textContent, sin innerHTML.
  • ✓Escala: 12 o 1.200 posts, el índice navegable pesa igual.

Arquitectura de código

La lógica algorítmica detrás de cada pieza.

Se usa Fuse.js, que implementa el algoritmo Bitap (a.k.a. shift-or / Baeza-Yates–Gonnet): una búsqueda aproximada que, mediante operaciones de bits, encuentra coincidencias del patrón dentro de un texto permitiendo hasta k errores (inserciones, borrados, sustituciones — esencialmente distancia de Levenshtein acotada). Fuse envuelve eso con pesos por campo y un score normalizado. 

fuse = new Fuse(data, {
  keys: [
    { name: 'title',       weight: 0.5 },  // el título manda
    { name: 'tags',        weight: 0.2 },
    { name: 'description', weight: 0.2 },
    { name: 'excerpt',     weight: 0.1 },
  ],
  threshold: 0.4,          // 0 = exacto · 1 = cualquier cosa
  ignoreLocation: true,    // el match puede estar en cualquier parte
  minMatchCharLength: 2,
});

const results = fuse.search(term); // [{ item, score, matches }]
¿Qué es exactamente threshold?›

Umbral de tolerancia del score de Fuse (0–1). 0.4 es el punto dulce: deja pasar erratas razonables (“dockr”, “astr”) pero descarta ruido. Subirlo encuentra más pero ensucia; bajarlo se vuelve casi exacto.

¿Por qué pesos por campo?›

Un acierto en el título (peso 0.5) debe rankear por encima de uno enterrado en el excerpt (0.1). Fuse combina las distancias por campo ponderándolas, así el orden de resultados refleja relevancia real.

¿Por qué lazy (import() dinámico)?›

Vite parte import('fuse.js') en un chunk separado. Solo se descarga cuando el usuario enfoca el buscador, no en la carga inicial. El 90% que solo navega no paga ese coste.

El índice JSON (endpoint estático)

// src/pages/[lang]/blog-search.json.ts
export const GET = async ({params}) => {
  const posts = getLangPosts(all, params.lang);
  const index = posts.map(p => ({
    title: p.data.title,
    description: p.data.description,
    tags: p.data.tags,
    url: `/${lang}/blog/${slug}/`,
    excerpt: strip(p.body).slice(0,160), // NO el body entero
  }));
  return Response.json(index);
};

Metadatos, no contenido. ~6 KB para 12 posts. Aquí está la clave del ahorro frente al modelo anterior.

La paginación es estática: se calcula en build time, no en el cliente. El número de páginas es Math.ceil(total / PAGE_SIZE); cada página corta su porción con slice(inicio, fin) donde inicio = (page-1)*PAGE_SIZE y fin = page*PAGE_SIZE. 

export const PAGE_SIZE = 9;

// Página 1 vive en /[lang]/blog
const totalPages = Math.ceil(posts.length / PAGE_SIZE);
const pagePosts  = posts.slice(0, PAGE_SIZE);

// Páginas 2..N → /[lang]/blog/page/[page].astro
export function getStaticPaths() {
  for (const lang of langs) {
    const posts = getLangPosts(all, lang);
    const total = Math.ceil(posts.length / PAGE_SIZE);
    for (let page = 2; page <= total; page++)
      paths.push({ params: { lang, page: String(page) } });
  }
}
const start = (currentPage - 1) * PAGE_SIZE;

¿Por qué /blog/page/N y no /blog/[...page]?›

Ya existe un catch-all [...slug] para los artículos. Un segundo rest-route en el mismo nivel sería ambiguo. Un segmento literal page/ elimina toda colisión y da URLs limpias.

¿Y cuando se busca? ¿Cómo muta el estado?›

La paginación gobierna solo el modo navegar. Al escribir, se ocultan rejilla y nav y se renderiza desde el índice fuzzy (todo el corpus). Al vaciar el input, vuelve la rejilla paginada en su página actual. Son dos modos disjuntos: nunca “se filtra solo lo visible”.
